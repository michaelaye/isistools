# CSM: how it gets established, how it gets verified, and how to add JANUS

Research notes investigating two questions:

1. How does the Community Sensor Model (CSM) standard get established and
   verified in the planetary community? Who says it's "better"?
2. How would we go about creating a new CSM sensor model for the JANUS
   camera on ESA's JUICE mission?

These notes are informal, source-cited, and intended to be a starting
point â€” not a definitive specification. Last updated 2026-04-11.

## 1. Origin and governance of CSM

CSM is **not** a planetary thing in origin. It is a US Department of
Defense / Intelligence Community standard, owned and maintained by the
**National Geospatial-Intelligence Agency (NGA)** through the
**Community Sensor Model Working Group (CSMWG)**, a chartered focus
group inside NGA's Geospatial Intelligence Standards Working Group (GWG).
The authoritative specification is the **CSM Technical Requirements
Document, NGA.STND.0017**. Version 3.0 was released in **November 2010**;
the spec went to **v3.0.3 in 2017**, and a **2018 API update** added
settable-radii functions â€” the change that finally made CSM usable for
non-Earth bodies (Mars, Moon, Mercury, etc).

The reference C++ headers live at <https://github.com/ngageoint/csm>.
CSM is C++, plugin-based: a "sensor exploitation tool" (SET, in NGA
parlance) loads CSM plugin shared libraries at runtime, and each plugin
contains one or more `csm::Model` subclasses that implement
`groundToImage()`, `imageToGround()`, and friends. The standard is
**not ISO**, **not OGC** (though OGC's "Name Type Specification â€” Sensor
Models and Parameters" registers CSM as an external sensor-model
registry entry). It is NGA-internal, with public spec PDF and public
header repo, but no formal standards-body status.

Sources:

- [NGA CSMWG governance landing page](https://gwg.nga.mil/gwg/focus-groups/Community_Sensor_Model_Working_Group_(CSMWG).html)
- [CSM TRD v3.0 PDF (NGA.STND.0017)](https://gwg.nga.mil/documents/csmwg/documents/CSM_TRD_Version_3.0__15_November_2010.pdf)
- [CSM reference headers (ngageoint/csm)](https://github.com/ngageoint/csm)
- [OGC Name Type Specification 18-042r4](https://docs.ogc.org/pol/18-042r4.html)

## 2. Relationship to ISIS / CSPICE camera models

CSM came **first**, by ~15 years. ISIS's native camera model framework
(`Isis::Camera` and per-instrument subclasses like `CTXCamera`,
`HiriseCamera`, `LrocNacCamera`, `MdisCamera`, â€¦) is built directly on
NAIF CSPICE kernels and dates back to the 1990s. Each mission has a
hand-written C++ subclass that knows how to read its instrument kernel
(IK) parameters and convert detector coordinates to body-fixed
coordinates via SPICE.

USGS Astrogeology Science Center pursued CSM compatibility around
2017â€“2020 â€” not because they thought CSM was more *accurate*, but
because:

1. **Commercial photogrammetry tools (BAE SOCET GXP)** could only consume
   CSM, blocking planetary users from modern stereo and bundle-adjustment
   workflows. The same applied to the open-source NASA Ames Stereo
   Pipeline (ASP), which switched to CSM as its preferred camera
   abstraction around the same time.
2. **The 2018 CSM API update** finally allowed non-Earth radii. Before
   that, planetary users were stuck shimming Earth-shaped CSM models or
   running legacy ISIS chains.
3. **A common API decouples sensor geometry from analytical software**.
   Once the camera projection is encapsulated as a CSM model and stored
   as a JSON Image Support Data (ISD) file, the same model can be
   consumed by ISIS, ASP, SOCET GXP, knoten, custom Python, and our
   own `csm2map` â€” without re-implementing the math each time.

The output of the USGS effort is a stack of three repos:

- **`DOI-USGS/usgscsm`** â€” C++ CSM plugin implementing four sensor model
  classes (frame, line scan, push frame, SAR). Distributed as
  `libusgscsm.dylib`/`.so` plus a `csmapi` Python wrapper generated by
  SWIG.
- **`DOI-USGS/ale`** â€” Python Abstraction Layer for Ephemerides. Reads
  ISIS cube labels and SPICE kernels (or ISIS-attached SPICE blobs) and
  emits the JSON ISD that `usgscsm` can construct a model from.
- **`DOI-USGS/knoten`** â€” convenience wrappers for stereo, jitter
  solving, and bundle adjustment using CSM models. (We dropped this
  dependency in `csm2map` because its conda packaging is broken on
  arm64.)

Critically: **CSPICE is not replaced**. SPICE kernels still drive
everything. ALE just reads them and re-emits the relevant numbers as
JSON, which `usgscsm` then parses. Our `csm2map` invokes
`ale.loads(cube)` â†’ ISD JSON â†’ `csmapi.Plugin.constructModelFromISD()` â†’
`UsgsAstroLineScanSensorModel` instance, and the ISIS CTX cube's
embedded NAIF tables are still where the spacecraft pointing comes from.

## 3. How is a planetary CSM sensor model verified?

There is **no formal certification program**. There is no NGA-hosted
test harness, no per-sensor "validation report" in the NASA technical
memo sense, and no review board that signs off on a new model.
Verification is done by the USGS team and the ASP team through three
informal methods:

1. **Round-trip self-consistency tests**. For a candidate sensor model,
   pick image points across the detector, run `imageToGround` â†’
   `groundToImage`, and check that the round-trip residual is below
   some threshold (typically <0.01 px). This proves the math is
   internally consistent. We did this on our CTX cube earlier in this
   project â€” see the `Forward-reverse round-trip test at the bottom edge`
   experiment in the conversation history; CSM's residuals were exactly
   zero to machine epsilon at every test point.

2. **Comparison against the legacy ISIS Camera class**. For each mission,
   the same image is processed twice: once with `Isis::Camera` (CSPICE
   path) and once with the new CSM model. Then `cam2map`-style
   map-projected outputs are compared pixel-for-pixel, or pointing
   residuals are compared at known ground control points. The de facto
   acceptance bar is **agreement below ~0.5 pixels**, with reported
   values often well under 0.1 px for framing cameras and a few tenths
   of a pixel for line scanners. Our own validation against ISIS
   `cam2map` on CTX (in `scripts/disagreement_analysis.md`) found
   100% of overlapping pixels within 0.01 DN, 99.95% coverage match,
   and a residual ~14â€“18 K disagreeing pixels at the line-scan
   time-domain edges â€” which is the "few tenths of a pixel" floor.

3. **End-to-end stereo and DEM differencing**. Process a stereo pair
   through the new CSM model in ASP, generate a DEM, and difference
   that DEM against one produced by the legacy ISIS chain. Spatial
   residuals of the difference DEM are reported as pointing accuracy.
   This is the methodology described on the
   [usgscsm wiki "CSM Stereo Testing Examples"](https://github.com/DOI-USGS/usgscsm/wiki/CSM-Stereo-Testing-Examples)
   and in Laura et al. 2024.

The `usgscsm` repo contains a `tests/` directory with `googletest`
unit tests against hand-validated reference values, but those are
basic sanity checks, not full sensor validation. The real validation
status of each sensor lives in commit history, the
[issue tracker](https://github.com/DOI-USGS/usgscsm/issues), and the
2020/2024 papers.

**As of early 2026 the validated sensor list (per the wiki and the
2020/2024 papers) is roughly**: HiRISE, CTX, MARCI (MRO), CaSSIS (TGO),
HRSC (Mars Express), MDIS NAC/WAC (MESSENGER), LRO NAC/WAC, Dawn FC,
THEMIS, Kaguya TC, Mini-RF (LRO), Magellan SAR, OSIRIS-REx MapCam,
Europa Clipper EIS NAC/WAC (in active development), Lucy LORRI/L'TES.

## 4. Who says it's "better"?

**Nobody claims CSM is more accurate than ISIS camera models.** The
consensus in Laura et al. 2020 and 2024 is that CSM is **equivalent in
accuracy** to the legacy `Isis::Camera` implementations when properly
validated â€” both ultimately consume the same SPICE kernels via the
same NAIF API. What CSM gets you is *interoperability*, which has
real, measurable downstream value:

- ASP can consume CSM models directly without ISIS dependency.
- SOCET GXP (commercial photogrammetry) becomes available to planetary
  data for the first time.
- Bundle adjustment and jitter solving can be done in CSM-native
  optimizers that work across all instruments uniformly.
- New tools like our own `csm2map` can be written in pure Python +
  scipy without linking to the giant ISIS C++ library, which is what
  enabled our **5â€“6Ã— speedup** over ISIS `cam2map` (the speedup is
  not from a better camera model â€” it's from skipping ISIS's
  ProcessRubberSheet quad-tree and writing modern vectorized scipy
  code).

So the right framing is: **CSM = same accuracy, much better
ergonomics**. Our project is a concrete example of that: we get
identical-to-ISIS DN values to within 0.01 DN on every pixel, while
running 5â€“6Ã— faster and producing GeoTIFF directly instead of ISIS
cubes that need conversion.

## 5. Path to JANUS (JUICE) â€” concrete walkthrough

**JANUS** is the framing camera on ESA's **JUICE** spacecraft, en route
to Jupiter's icy moons Ganymede, Europa, and Callisto for the
2031â€“2034 prime mission. It is a **2000 Ã— 2000 pixel CCD framing
camera** with a 13-position filter wheel, mounted on the JUICE
spacecraft bus. SPICE instrument code: `JUICE_JANUS` (NAIF ID
**âˆ’28200**, per `juice_janus_v06.ti`).

The good news: **JANUS is a frame camera**, which is the simplest case
in CSM terms. There is already a generic `UsgsAstroFrameSensorModel` in
`usgscsm`, and several existing ALE drivers for similar instruments
(MESSENGER MDIS, Dawn FC, Mars Express SRC, Europa Clipper EIS NAC/WAC
in framing mode). **No new C++ class is needed** â€” the implementation
work is almost entirely on the ALE side, writing a Python "driver" that
knows how to read JANUS-specific metadata from a JUICE cube label.

### 5.1 What already exists

`usgscsm` provides five concrete sensor model classes
(`include/usgscsm/`):

| Class | Camera type | Use for |
|---|---|---|
| `UsgsAstroFrameSensorModel` | framing / snapshot | **JANUS**, MDIS, Dawn FC, OSIRIS-REx MapCam, Clipper EIS framing |
| `UsgsAstroLsSensorModel` | line scanner (push-broom) | HiRISE, CTX, HRSC, LROC NAC |
| `UsgsAstroPushFrameSensorModel` | push-frame (framelets) | LROC WAC, THEMIS |
| `UsgsAstroSarSensorModel` | synthetic aperture radar | Magellan, Mini-RF |
| `UsgsAstroProjectedSensorModel` | wrapper for already-projected imagery | post-processing |

For JANUS, the answer is `UsgsAstroFrameSensorModel`, full stop.

`ale` uses a mixin-based driver pattern. A driver class composes:

- A **sensor type mixin** from `ale/base/type_sensor.py`: `Framer`,
  `LineScanner`, `PushFrame`, `RollingShutter`, `Radar`. For JANUS:
  `Framer` (which sets `name_model = "USGS_ASTRO_FRAME_SENSOR_MODEL"`).
- A **label reader mixin** from `ale/base/label_*.py`: `IsisLabel` if
  the cube has been ingested into ISIS3, `Pds3Label` for raw PDS3
  products. For JANUS we want both eventually, but `IsisLabel` first
  because the rest of the planetary stack assumes ISIS cubes.
- A **SPICE source mixin** from `ale/base/data_naif.py` /
  `ale/base/data_isis.py`: `NaifSpice` for live kernels, `IsisSpice`
  for kernel blobs embedded in the cube. Both should be supported.
- A **distortion mixin** from `ale/base/type_distortion.py`:
  `NoDistortion`, `RadialDistortion`, etc. JANUS has a real radial
  distortion model â€” start with `NoDistortion` for v1, swap to
  `RadialDistortion` once you have coefficients from the JANUS team.
- The `Driver` base from `ale/base/base.py`.

A concrete driver overrides only what differs from the defaults:
`instrument_id`, `spacecraft_name`, `sensor_model_version`, `ikid`
(NAIF instrument frame code), `fikid` (filter NAIF code â€” important
for JANUS's 13-filter wheel), `focal_length`, `detector_center_sample/
line`, `pixel_size`, `focal2pixel_samples/lines`,
`ephemeris_start_time`, `exposure_duration`, and the relevant
`usgscsm_distortion_model` keys.

### 5.2 The closest existing template: Europa Clipper EIS framing

The single best template to copy is `ClipperEISWACFCIsisLabelNaifSpiceDriver`
in [`ale/drivers/clipper_drivers.py`](https://github.com/DOI-USGS/ale/blob/main/ale/drivers/clipper_drivers.py).
It's an icy-moon orbiter framing camera with a multi-filter wheel,
which is exactly the JANUS topology. Class structure:

```python
class ClipperEISWACFCIsisLabelNaifSpiceDriver(
    Framer, IsisLabel, NaifSpice, NoDistortion, Driver
):
    @property
    def instrument_id(self):
        return "EUROPAM_EIS_WAC"

    @property
    def fikid(self):
        return EIS_WAC_FILTER_CODES[self.filter_name]

    @property
    def filter_name(self):
        return self.label['IsisCube']['BandBin']['FilterName']

    @property
    def focal_length(self):
        return 46.25  # mm â€” hardcoded per camera

    @property
    def detector_center_sample(self):
        return 4096 / 2

    @property
    def detector_center_line(self):
        return 2048 / 2

    @property
    def detector_start_line(self):
        return EIS_FILTER_START_LINES[self.filter_name]

    @property
    def detector_start_sample(self):
        return EIS_FILTER_START_SAMPLES[self.filter_name]

    @property
    def focal2pixel_lines(self):
        return [0.0, 0.0, 1.0 / 0.014]  # 14 Âµm pixel pitch

    @property
    def focal2pixel_samples(self):
        return [0.0, -1.0 / 0.014, 0.0]
```

with the per-filter lookup tables defined as module-level dicts:

```python
EIS_WAC_FILTER_CODES = {"CLEAR": -159141, "NUV": -159145, ...}
EIS_FILTER_START_LINES = {"CLEAR": 580, "NUV": 415, ...}
EIS_FILTER_START_SAMPLES = {"CLEAR": 132, "NUV": 32, ...}
```

That's it. ~140 lines per driver class, one file. The whole
`clipper_drivers.py` is 686 lines covering four variants (NAC/WAC Ã—
framing/pushbroom).

### 5.3 What you'd actually do for JANUS

Assuming ISIS3 already supports ingesting JUICE/JANUS images into
`.cub` files (this is the **hard prerequisite** â€” see Â§5.4 below):

1. **Create `ale/drivers/juice_drivers.py`** by copying
   `clipper_drivers.py` and renaming. Strip `RollingShutter` (JANUS is
   not a rolling shutter), rename classes to e.g.
   `JuiceJanusIsisLabelNaifSpiceDriver` and
   `JuiceJanusPds3NaifSpiceDriver`.
2. **Replace the NAIF constants** with values from
   [`juice_janus_v06.ti`](https://spiftp.esac.esa.int/data/SPICE/JUICE/kernels/ik/juice_janus_v06.ti):
   `ikid = -28200`, focal length, pixel size, detector center, optical
   distortion model.
3. **Build the per-filter `JANUS_FILTER_CODES` lookup table** from the
   IK file â€” JANUS has 13 filters, each with its own NAIF body code
   following the pattern `-28200X` or `-2820XX`. Read the IK to
   confirm the exact codes.
4. **Register the new driver** in `ale/drivers/__init__.py`. Add a
   `JUICE_DRIVERS` list and append it to the priority-ordered driver
   discovery iteration in `AleJsonEncoder` / `ale.load`.
5. **Add a SpiceQL config for JUICE** (or have ALE consumers pass
   `props={"kernels": [...meta-kernel...]}` explicitly). SpiceQL
   doesn't yet ship a JUICE config â€” this is a small TOML file that
   tells SpiceQL where to find ESA's JUICE kernels.
6. **Test fixtures**: add `tests/pytests/data/isds/juice_janus_isd.json`
   (a hand-validated golden ISD) and `tests/pytests/test_juice_drivers.py`
   that does `compare_dicts(ale.loads(label), expected)`.
7. **Distortion**: leave as `NoDistortion` for v1. Once you have real
   coefficients from the JANUS team, swap to a `RadialDistortion`
   mixin and add the `usgscsm_distortion_model` keys in the driver.
8. **`usgscsm` itself: change nothing**. Confirm by feeding the
   generated ISD JSON to `usgscsm_cam_test` (a CLI tool that ships
   with usgscsm) â€” if it constructs a `RasterGM` and round-trips
   successfully, you're done.
9. **Open a PR** against `DOI-USGS/ale`. Reviewer wants:
   round-trip test against the embedded ISIS BLOBs of a real JUICE
   image (or against a test fixture you constructed), passing CI.

### 5.4 The hard prerequisite: ISIS3 has to support JUICE first

ALE and `usgscsm` work *on top of* ISIS-ingested cubes. This means:

- **`juice2isis`** ingestion command needs to exist in
  `DOI-USGS/ISIS3`, mirroring how `mroctx2isis` ingests CTX or
  `mdis2isis` ingests MDIS. Without this, you can't get a JUICE PDS
  label into a `.cub` file with proper NAIF kernels attached.
- **`isis/src/juice/objs/JuiceJanusCamera/`** â€” a new C++ subclass of
  `Isis::FramingCamera` that knows how to read JUICE-specific kernel
  parameters. Modeled on `isis/src/messenger/objs/MdisCamera/`.
- **`$ISIS3DATA/juice/kernels/`** â€” a kernel database directory with
  JUICE FK, IK, SCLK, SPK, CK organized in the ISIS conventional
  layout, plus IAK (Instrument Addendum Kernel) text files containing
  any keys ALE expects that ESA's IK doesn't ship.

I checked GitHub via `gh search`: as of early 2026 there is **zero
JUICE/JANUS support in any DOI-USGS repository** (`ISIS3`, `ale`, or
`usgscsm`). No issues, no PRs, no draft branches. You'd be the first.
Worth cross-checking the OpenPlanetary community and any internal
USGS Astrogeology trackers in case prototypes exist outside DOI-USGS.

### 5.5 ESA SPICE for JUICE

ESA's SPICE Service publishes the JUICE kernel set at:

- <https://spiftp.esac.esa.int/data/SPICE/JUICE/kernels/>
- Mirror at <https://naif.jpl.nasa.gov/pub/naif/JUICE/kernels/>

Specifically the relevant files for JANUS:

- `fk/juice_v26.tf` â€” frames kernel (current as of 2026-04)
- `ik/juice_janus_v06.ti` â€” JANUS instrument kernel with focal
  length, distortion model, FOV, and per-filter NAIF codes
- `sclk/juice_fict_v0X.tsc` â€” spacecraft clock (one for cruise, one
  for prime mission)
- `pck/`, `lsk/`, `spk/`, `ck/` â€” body shapes, leap seconds,
  ephemerides, and orientation tables

The full `juice` meta-kernel that would furnish all of these is
documented in the ESA SPICE Service for JUICE landing page at
<https://www.cosmos.esa.int/web/spice/spice-for-juice>.

### 5.6 Validation of a JANUS CSM model

Before any merged PR, you'd want to run the same three tests USGS
applies to every other sensor:

1. **Round-trip self-consistency**: pick test points across the
   2000Ã—2000 detector and confirm `imageToGround` followed by
   `groundToImage` returns the same coordinates to <0.01 px. This
   proves the math is internally consistent.
2. **Compare to a JANUS ISIS Camera class** if one exists. If not (and
   right now it doesn't), this step is skipped or replaced with a
   manual ground control point check using known surface features on
   Ganymede or Europa.
3. **Stereo DEM check**: process a real or simulated JANUS stereo pair
   through ASP using your new CSM model, generate a DEM, and check
   the residuals against any reference DEM (limited for the JUICE
   targets, but Ganymede has the Voyager-era global topography from
   stereo and limb fitting).

The "validation report" for JANUS would then be a short ESS or LPSC
abstract describing the methodology and the residuals â€” same
deliverable shape as Laura et al. 2020/2024.

## 6. Take-aways

- CSM is an NGA-owned C++ plugin standard from 2010 (v3.0), with the
  2018 API update unlocking planetary use.
- USGS built `usgscsm` + `ale` from 2017â€“2020 to give the planetary
  community a CSM-compliant alternative to ISIS's CSPICE camera
  models, primarily for *tooling interoperability*, not for accuracy
  gains.
- Validation is informal: round-trip tests, comparison vs ISIS Camera,
  and stereo-DEM differencing. Sub-pixel agreement (<0.5 px, often
  <0.1 px) is the de facto bar.
- Our `csm2map` validation against ISIS `cam2map` on MRO CTX
  (99.95% coverage match, 100% within 0.01 DN, 5â€“6Ã— faster) is a
  textbook example of the "same accuracy, much better ergonomics"
  pitch.
- For JANUS, the work is mostly in `ale` â€” write a new driver
  modelled on Europa Clipper EIS framing or MESSENGER MDIS, ~140
  lines of Python, zero C++. The hard prerequisite is **ISIS3
  support for JUICE ingestion**, which doesn't exist anywhere in
  the DOI-USGS stack as of early 2026. Adding `juice2isis` and a
  `JuiceJanusCamera` C++ class is the actual blocker, not the CSM
  side.
